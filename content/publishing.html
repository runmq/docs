Core · Publishing

Publishing messages.

Publishing is intentionally minimal — pick a topic, hand it a payload, and RunMQ takes care of routing, durability, and delivery.

The publish API

mq.publish(topic: string, message: Record<string, any>, correlationId?: string): void;
  • topic — the topic you're publishing to. Every processor subscribed to this topic receives the message.
  • message — your payload. Make sure it satisfies any schemas the consumers expect.
  • correlationId — optional. Useful for tracing a message across services.

Examples

// Simple publish
mq.publish('user.created', {
  userId: 'u_123',
  email: 'user@example.com',
  name: 'John Doe',
});

// With a correlation id for distributed tracing
mq.publish('order.placed', orderPayload, requestContext.traceId);

Atomic fan-out

You publish once. RabbitMQ delivers a copy to every subscribed processor's queue, atomically. There is no application-level fan-out and no risk of "delivered to A but not B".

mq.publish('user.created', {...}) │ ▼ Topic exchange (user.created) ├──► Queue: emailService ├──► Queue: analyticsService └──► Queue: notificationService

Publish ordering & durability

  • Messages are published as persistent — they survive RabbitMQ restarts.
  • Within a single channel, ordering is preserved.
  • Across consumers, ordering depends on prefetch + concurrency. Use consumersCount: 1 if strict ordering matters per processor.

Common mistakes

Don't publish before RunMQ.start() resolves. Publish calls before the connection is ready will throw Error('RunMQ is not connected'). Wait for start, then publish from request handlers or jobs.
No processor? No subscriber. A topic with no processor means messages have nowhere to go. RunMQ does not buffer them in your app — they're routed by RabbitMQ and dropped if no queue is bound.

Tracing with correlation IDs

Pass a correlation ID at publish time and RunMQ surfaces it on the consumer side via message.meta.correlationId. Wire it through your logger and you've got end-to-end tracing for free. 

// Publisher
mq.publish('email.send', { to, subject, body }, ctx.traceId);

// Consumer
await mq.process('email.send', { name: 'emailWorker', attempts: 3 }, async (msg) => {
  logger.info({ traceId: msg.meta.correlationId }, 'sending email');
  await transport.send(msg.message);
});
