Event bus for event-driven architecture.

Publish a fact once. Many services react to it independently — each with its own queue, retries, and DLQ. The event bus is the backbone of decoupled microservices: services stop calling each other and start reacting to what's happening in the system.

What it is

An event bus is the publish/subscribe pattern applied at the architecture level. A producer announces something that happened — "a user signed up", "an order was placed", "an invoice was paid" — and any number of consumers can subscribe to that fact and do their own work in response. The producer doesn't know who listens. The listeners don't know about each other.

Publisher ─► topic: user.created ├─► queue: emailService → DLQ: emailService_dlq ├─► queue: analyticsService → DLQ: analyticsService_dlq └─► queue: notificationService → DLQ: notificationService_dlq

Critically, this is not application-level fan-out. The broker copies the message to every bound queue atomically. There is no in-process loop, no risk of "delivered to A but not B", and no shared state between consumers.

Events vs. commands

Before going further, it's worth being precise about what an event is. The shape of the message — and the way systems use it — is different from a command (which most synchronous APIs implicitly are):

CommandEvent
TenseImperative — SendEmailPast — UserCreated
Intent"Do this""This happened"
RecipientsExactly oneZero or more
Producer knows consumer?YesNo
Failure modelProducer cares about the outcomeProducer is done; consumers own their outcomes

Background processing usually carries commands (email.send = "send an email"). An event bus carries events (user.created = "a user was created — react however you like"). The distinction shapes how you name topics and where responsibility lives.

Why it matters in microservices

Microservices shine when services can change, deploy, and fail independently. The biggest threat to that independence is synchronous coupling — service A reaching into service B's API. The more direct calls between services, the more your "microservices" behave like a distributed monolith with extra latency.

An event bus replaces synchronous coupling with three properties that microservice teams care about deeply:

  • Temporal decoupling. Producer and consumers don't have to be online at the same time. The user-service can publish user.created at 3am; the analytics consumer can be down for maintenance and pick up the backlog when it returns. No retries-in-the-request, no 502s on the user signup endpoint.
  • Spatial decoupling. The producer doesn't know who consumes its events. Add a new fraud-detection service next quarter and it just subscribes — no PR against the user-service, no version bump, no shared library.
  • Failure containment. A bug in the analytics consumer parks bad messages in the analytics DLQ. The email consumer keeps shipping. Compare with synchronous chains, where one slow service degrades every caller upstream.
The architectural payoff: teams can ship features without coordination. New consumers attach themselves to existing events. Old consumers don't need to know they exist. The wiring lives in the broker, not in your codebase.

Where it shines

ScenarioWhy an event bus fits
User signup with downstream side-effects Welcome email, CRM sync, analytics, notifications — none of which the auth service should know about.
Order lifecycle (placed → paid → shipped) Inventory, billing, fulfillment, and notifications all react to the same state transitions.
Audit logging across services One audit consumer subscribes to many events. Adding audit coverage doesn't touch business code.
Search indexing The search service reacts to write events from many bounded contexts; the writers know nothing about search.
Cache invalidation Caches subscribe to mutation events. Writers don't carry a list of caches to bust.
Materialized views / read models (CQRS) Read-side projections rebuild themselves by subscribing to write-side events.
Saga / process orchestration Long-running workflows advance step by step on event arrivals.

The minimum viable setup

import { RunMQ } from 'runmq';

const mq = await RunMQ.start({ url: 'amqp://localhost:5672' });

// Email — short retries, low concurrency
await mq.process('user.created', {
  name: 'emailService',
  consumersCount: 2,
  attempts: 3,
  attemptsDelay: 2_000,
}, async (msg) => sendWelcomeEmail(msg.message));

// Analytics — many retries, OK to fall behind
await mq.process('user.created', {
  name: 'analyticsService',
  consumersCount: 4,
  attempts: 10,
  attemptsDelay: 5_000,
}, async (msg) => track('user_created', msg.message));

// Notifications — strict schema, every message matters
await mq.process('user.created', {
  name: 'notificationService',
  consumersCount: 2,
  attempts: 5,
  attemptsDelay: 1_000,
  messageSchema: {
    type: 'ajv',
    failureStrategy: 'dlq',
    schema: { /* … */ },
  },
}, async (msg) => pushNotification(msg.message));

// Anywhere in the producer service:
mq.publish('user.created', { userId: 'u_123', email: 'a@b.com' });

Three independent consumers, three independent failure modes, three independent tuning knobs. One publish.

Designing good events

Events are a public API between services — arguably more public than your HTTP endpoints, because consumers depend on the semantics of what happened, not just the wire format. Treat them with the same care.

Name them in past tense, by domain

user.created, order.placed, invoice.paid. Not create_user, not send_email. The name is a fact about the world, not a request to do something.

Carry just enough payload

Two viable patterns, with very different trade-offs:

StylePayloadTrade-off
Notification event Just identifiers — { userId: 'u_123' } Tiny messages; consumers fetch fresh data. But adds a request back to the producer service — partial coupling.
Event-carried state transfer The full relevant state — { userId, email, plan, … } Consumers are autonomous; producer can be down. But payloads grow, and stale data is possible if consumers process slowly.

In practice, most teams settle on event-carried state transfer with a small, stable subset of fields. Add a schemaVersion early so you can evolve.

Version the schema

mq.publish('user.created', {
  schemaVersion: 1,
  userId: 'u_123',
  email: 'a@b.com',
  plan: 'free',
});

Adding fields is safe. Renaming or removing fields is a breaking change — bump the version (or publish to a new topic like user.created.v2) and let consumers migrate at their pace.

Tuning per consumer, not per topic

The point of the event bus is that consumers are independent — and so is their tuning. RunMQ makes this concrete: every processor sets its own concurrency, retry budget, delay, and schema. A single user.created event can flow into:

  • An email consumer with 2 concurrent workers and 3 fast retries (the SMTP relay is usually fine).
  • A billing consumer with 1 worker and 10 patient retries (the payment provider has SLAs).
  • An analytics consumer with 8 workers and many retries (eventual consistency is fine).

None of those tunings affect the others. None of those failures affect the others.

Idempotency is non-negotiable

Every consumer in an event bus must be idempotent. RunMQ retries on failure, RabbitMQ may redeliver if a consumer crashes mid-ack, and the same logical event may reach two replicas of the same consumer. Bake idempotency into the handler: 

await mq.process('order.placed', { name: 'inventory', attempts: 5 }, async (msg) => {
  const { orderId, items } = msg.message;
  if (await db.inventoryReservations.exists(orderId)) return;       // already reserved
  await db.inventoryReservations.reserve(orderId, items);
});

Use a stable key from the event (orderId, an explicit eventId, etc.) and the same database transaction that does the work to record "I handled this". If you can't co-transact, use an outbox or a deduplication store. Don't assume "exactly once".

Operational habits

  • Name queues by service, not by event. emailService, not user.created.queue. A service may subscribe to many events; its queue is "stuff this service is processing".
  • Wire one DLQ per service, per event. Having a dedicated DLQ for each processor lets you triage failures without mixing concerns. RunMQ does this for you.
  • Monitor queue depth, not just message rate. Depth tells you whether consumers are keeping up with producers. A growing queue is a warning before it's an outage.
  • Treat the broker topology as code. RunMQ asserts queues, exchanges, and bindings on startup, so the runtime configuration is the source of truth.

Anti-patterns

Don'tWhy
Encode commands as events send_email_event is a command in disguise — exactly one consumer, the producer cares about the result. Use a background-processing pattern instead.
Make consumers depend on each other's order Events arrive in roughly the order published, but not strictly. If "order shipped" must follow "order paid", that's a saga, not two unrelated events.
Fan out in application code If the producer publishes the same event N times to N topics, you've reinvented the broker — badly. Publish once; let bindings do the rest.
Couple consumers to producer internals An event payload is an API. Don't dump your ORM model directly. Pick a deliberate shape and version it.
Treat events as logs The bus is for reacting, not auditing. If you want a permanent record, persist it in a dedicated store (often via an audit consumer).

Choreography vs. orchestration

An event bus naturally favors choreography — each service reacts to events on its own. That's powerful, but for multi-step workflows (place order → reserve inventory → charge card → notify warehouse), pure choreography can scatter the logic across services and make debugging painful.

When a workflow has clear steps and compensating actions on failure, model it as an explicit saga — a coordinator service that listens for step-completed events and publishes the next command. The coordinator owns the state machine; the participants stay simple. RunMQ's primitives support either style; the choice is architectural, not technical.

Coming from background processing

If you started with the background processing pattern, growing into an event bus costs you nothing. Add a second processor with a different name to the same topic, and you've crossed the line: 

// You already had this:
await mq.process('order.placed', { name: 'fulfillment', attempts: 5 }, fulfill);

// Adding analytics doesn't touch the fulfillment service:
await mq.process('order.placed', { name: 'analytics',  attempts: 10 }, track);

No migration. No coordination. No new infrastructure. The same publish now reaches two independent consumers — and the next team that wants in just adds a third.

An event bus isn't a feature you turn on — it's the architecture that emerges when you let services react to facts instead of calling each other.
