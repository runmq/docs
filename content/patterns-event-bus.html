Patterns

Event bus for event-driven architecture.

Publish a fact once. Many services react to it independently — each with its own queue, retries, and DLQ. The backbone of decoupled microservices: services stop calling each other and start reacting to what's happening in the system.

Quick start

import { RunMQ } from 'runmq';

const mq = await RunMQ.start({ url: 'amqp://localhost:5672' });

// Email — short retries, low concurrency
await mq.process('user.created', {
  name: 'emailService',
  consumersCount: 2,
  attempts: 3,
  attemptsDelay: 2_000,
}, async (msg) => sendWelcomeEmail(msg.message));

// Analytics — many retries, OK to fall behind
await mq.process('user.created', {
  name: 'analyticsService',
  consumersCount: 4,
  attempts: 10,
  attemptsDelay: 5_000,
}, async (msg) => track('user_created', msg.message));

// Notifications — strict schema, every message matters
await mq.process('user.created', {
  name: 'notificationService',
  consumersCount: 2,
  attempts: 5,
  attemptsDelay: 1_000,
  messageSchema: {
    type: 'ajv',
    failureStrategy: 'dlq',
    schema: { /* … */ },
  },
}, async (msg) => pushNotification(msg.message));

// Anywhere in the producer service:
mq.publish('user.created', { userId: 'u_123', email: 'a@b.com' });

Three independent consumers, three independent failure modes, three independent tuning knobs. One publish.

Designing good events

Name them in past tense, by domain

user.created, order.placed, invoice.paid. Not create_user, not send_email. The name is a fact about the world, not a request to do something.

Carry just enough payload

StylePayloadTrade-off
Notification event Just identifiers — { userId: 'u_123' } Tiny messages; consumers fetch fresh data. But adds a request back to the producer — partial coupling.
Event-carried state transfer Full relevant state — { userId, email, plan, … } Consumers stay autonomous. But payloads grow and stale data is possible if consumers process slowly.

Most teams settle on event-carried state transfer with a small, stable subset of fields.

Version the schema

mq.publish('user.created', {
  schemaVersion: 1,
  userId: 'u_123',
  email: 'a@b.com',
  plan: 'free',
});

Adding fields is safe. Renaming or removing fields is breaking — bump the version (or publish to user.created.v2) and let consumers migrate at their pace.

Tuning per consumer

Every processor sets its own concurrency, retry budget, delay, and schema. A single user.created event can flow into:

  • An email consumer with 2 workers and 3 fast retries.
  • A billing consumer with 1 worker and 10 patient retries.
  • An analytics consumer with 8 workers and many retries.

None of those tunings affect the others. None of those failures affect the others.

Idempotency is non-negotiable

Every consumer must be idempotent. RunMQ retries on failure, RabbitMQ may redeliver if a consumer crashes mid-ack, and the same logical event may reach two replicas of the same consumer. 

await mq.process('order.placed', { name: 'inventory', attempts: 5 }, async (msg) => {
  const { orderId, items } = msg.message;
  if (await db.inventoryReservations.exists(orderId)) return;       // already reserved
  await db.inventoryReservations.reserve(orderId, items);
});

Use a stable key from the event (orderId, an explicit eventId) and the same transaction that does the work to record "I handled this". Don't assume "exactly once".

Operational habits

  • Name processors by service + event. A service that subscribes to one event can use the bare service name (emailService); a service that subscribes to several should disambiguate per processor — emailService.userCreated, emailService.orderShipped. Pick one separator and stick with it. Avoid naming by event alone — two services on the same event would clash.
  • Monitor queue depth, not just message rate. Depth tells you whether consumers are keeping up. A growing queue is a warning before it's an outage.
  • Watch DLQ depth too. Every processor gets its own DLQ automatically — alert on a non-zero one.
  • Treat the broker topology as code. RunMQ asserts queues, exchanges, and bindings on startup, so the runtime configuration is the source of truth.

Anti-patterns

Don'tWhy
Encode commands as events send_email_event is a command in disguise — exactly one consumer, producer cares about the result. Use background processing.
Make consumers depend on each other's order Events arrive in roughly the order published, but not strictly. Multi-step ordering is a saga, not two unrelated events.
Fan out in application code Publishing the same event N times to N topics reinvents the broker — badly. Publish once; let bindings do the rest.
Couple consumers to producer internals An event payload is an API. Don't dump your ORM model directly. Pick a deliberate shape and version it.
Treat events as logs The bus is for reacting, not auditing. Persist a permanent record in a dedicated store (often via an audit consumer).

Deep dive: how the pattern works

What an event bus is

Pub/sub at the architecture level. A producer announces something that happened — "a user signed up", "an order was placed" — and any number of consumers subscribe and react. The producer doesn't know who listens; the listeners don't know about each other.

Publisher ─► topic: user.created ├─► queue: emailService → DLQ: emailService_dlq ├─► queue: analyticsService → DLQ: analyticsService_dlq └─► queue: notificationService → DLQ: notificationService_dlq

This is not application-level fan-out. The broker copies the message to every bound queue atomically — no in-process loop, no risk of "delivered to A but not B".

Events vs. commands

CommandEvent
TenseImperative — SendEmailPast — UserCreated
Intent"Do this""This happened"
RecipientsExactly oneZero or more
Producer knows consumer?YesNo

Background processing usually carries commands. An event bus carries events. The distinction shapes how you name topics and where responsibility lives.

Why it matters in microservices

The biggest threat to microservice independence is synchronous coupling — service A reaching into service B's API. The more direct calls, the more your "microservices" behave like a distributed monolith.

  • Temporal decoupling. Producer and consumers don't need to be online at the same time. The analytics consumer can be down for maintenance and pick up the backlog when it returns.
  • Spatial decoupling. The producer doesn't know who consumes its events. Add a fraud-detection service next quarter and it just subscribes — no PR against the producer, no shared library.
  • Failure containment. A bug in one consumer parks bad messages in its DLQ. Other consumers keep shipping.

Where it shines

ScenarioWhy an event bus fits
User signup with downstream side-effectsEmail, CRM, analytics, notifications — none of which the auth service should know about.
Order lifecycle (placed → paid → shipped)Inventory, billing, fulfillment, and notifications all react to state transitions.
Audit loggingOne audit consumer subscribes to many events without touching business code.
Search indexing & cache invalidationIndex/cache services subscribe to write events; writers know nothing about them.
Materialized views (CQRS)Read-side projections rebuild themselves by subscribing to write-side events.

Choreography vs. orchestration

An event bus naturally favors choreography — each service reacts on its own. For multi-step workflows (place order → reserve inventory → charge card → notify warehouse), pure choreography can scatter logic and make debugging painful. When a workflow has clear steps and compensating actions on failure, model it as a saga — a coordinator that listens for step-completed events and publishes the next command. RunMQ's primitives support either style.

Coming from background processing

If you started with the background processing pattern, growing into an event bus costs nothing. Add a second processor with a different name to the same topic: 

// You already had this:
await mq.process('order.placed', { name: 'fulfillment', attempts: 5 }, fulfill);

// Adding analytics doesn't touch the fulfillment service:
await mq.process('order.placed', { name: 'analytics',  attempts: 10 }, track);
An event bus isn't a feature you turn on — it's the architecture that emerges when you let services react to facts instead of calling each other.
