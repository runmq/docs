RunMQ vs BullMQ — the numbers.

A fair, reproducible benchmark comparing RunMQ (RabbitMQ) and BullMQ (Redis) across six performance dimensions. Tested on MacBook Pro M4 Max, Docker Desktop, mean of 3 runs ± stddev.

Transparency. The benchmark was written by Claude (Anthropic) at the request of the RunMQ maintainer, with explicit instructions to optimize BullMQ to the maximum (addBulk(), skipStalledCheck, skipLockRenewal, etc.). Both libraries got equal Docker resources (2 CPU, 4 GB RAM) and identical settling time. The two systems have fundamentally different architectures, so see Known caveats for what these numbers really mean.

Throughput at a glance

Publish throughput (msg/s)

What this measures: how many messages per second a single producer can push, with no consumer attached. Each library publishes a fixed total (10,000 messages, 100-byte payload) and the elapsed time gives msg/s.

4.7× faster. RunMQ publishes via in-process AMQP buffer; BullMQ awaits a Redis pipeline round-trip.

Consume throughput (msg/s)

What this measures: how fast a single no-op consumer drains a pre-loaded queue. The queue is filled with 10,000 messages, then a consumer with a no-op handler is started; timing runs from the first message consumed to the last.

3.3× faster.

Concurrent consumers (msg/s, 1ms simulated work)

What this measures: drain rate when handlers do non-trivial work — each handler awaits a 1 ms sleep before acking. Two configurations: one in-flight message at a time (· 1) and eight in-flight per process (· 8). This isolates how well each library overlaps I/O across concurrent in-flight messages.

End-to-end latency

What this measures: per-message time from publish() returning to the consumer's handler receiving that same message. The producer stamps each message with a high-resolution timestamp; the consumer subtracts it from process.hrtime() on receipt. Numbers below are over 10,000 messages, mean of 3 runs ± stddev.

Payload size impact

What this measures: how publish and consume throughput change as the message payload grows. Same scenarios as the throughput tests above (publish: producer alone; consume: drain a pre-loaded queue), repeated at three payload sizes. All numbers are messages per second.

Both libraries slow down as payloads grow — serialization and network bytes dominate at 10 KB. The gap between RunMQ and BullMQ narrows at larger sizes because raw transport speed matters less when each message takes longer to move — though RunMQ is still 3×–6× faster across every size tested.

Reliability overhead

What this measures: consume throughput with retries disabled vs. enabled (3 attempts, 100 ms delay between attempts). Same single-consumer drain scenario as the consume-throughput test; the only difference is whether retry/DLQ machinery is configured. Tells you the price of opting into reliability when nothing actually fails.

ModeRunMQ (msg/s)BullMQ (msg/s)Ratio
Basic (no retries)29,187 ±2228,334 ±433.5×
With retries enabled29,700 ±4358,476 ±403.5×

Retries are essentially free in RunMQ — TTL-based requeue keeps the hot path untouched.

Known caveats

  • Confirmed publishes by default. RunMQ's publish() awaits a broker ack out of the box (publisher confirms enabled by default), measuring "confirmed persistence speed" — the same guarantee BullMQ's addBulk() provides via its Redis pipeline confirmation. Under that confirmed mode RunMQ measures ~56,400 msg/s — roughly 80% above BullMQ's ~32,200 msg/s on the same hardware. If you opt out via usePublisherConfirms: false, RunMQ becomes a buffer write (no delivery guarantee) and the headline number climbs further, but the comparison stops being apples-to-apples.
  • Execution ordering is fixed. RunMQ runs first in every scenario, paying the JIT cold-start cost. This biases against RunMQ.
  • 1 ms simulated work in concurrent-consumer tests over-weights per-message overhead. Realistic 10–100 ms work would narrow the gap.
