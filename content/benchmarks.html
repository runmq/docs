RunMQ vs BullMQ — the numbers.

A fair, reproducible benchmark comparing RunMQ (RabbitMQ) and BullMQ (Redis) across six performance dimensions. Tested on MacBook Pro M4 Max, Docker Desktop, mean of 3 runs ± stddev.

Transparency. The benchmark was written by Claude (Anthropic) at the request of the RunMQ maintainer, with explicit instructions to optimize BullMQ to the maximum (addBulk(), skipStalledCheck, skipLockRenewal, etc.). Both libraries got equal Docker resources (2 CPU, 4 GB RAM) and identical settling time. The two systems have fundamentally different architectures, so see Known caveats for what these numbers really mean.

Throughput at a glance

Publish throughput (msg/s)

4.7× faster. RunMQ publishes via in-process AMQP buffer; BullMQ awaits a Redis pipeline round-trip.

Consume throughput (msg/s)

3.3× faster. Single no-op consumer, measured first-consumed → last-consumed.

Known caveats

  • Different guarantees on publish. RunMQ's publish() writes to an in-process AMQP buffer; BullMQ's addBulk() awaits a Redis pipeline confirmation. Same scenario, different semantics — RunMQ is measuring "buffer write speed," BullMQ "confirmed persistence speed."
  • Redis is in-memory; RabbitMQ is disk-backed. RabbitMQ persists messages by default; Redis doesn't. This favors BullMQ on E2E latency by design.
  • Execution ordering is fixed. RunMQ runs first in every scenario, paying the JIT cold-start cost. This biases against RunMQ.
  • 1 ms simulated work in concurrent-consumer tests over-weights per-message overhead. Realistic 10–100 ms work would narrow the gap.
  • RunMQ's internal processor chain. RunMQ allocates 6 processor objects per message (deserializer, retries checker, acknowledger, etc.) — included in the consume numbers, biasing slightly against RunMQ.

git clone https://github.com/runmq/benchmark

./run.sh   # builds Docker images, runs benchmarks, opens results/report.html
