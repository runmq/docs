Quick start

Up and running in five minutes.

Install the package, connect to RabbitMQ, declare a processor, publish a message. That's the whole loop. Everything else in these docs is about tuning — not getting started.

Prerequisites

  • Node.js 18+
  • A RabbitMQ instance — local Docker, managed cloud, or otherwise.
  • The RabbitMQ management plugin enabled (recommended — unlocks dynamic retry tuning).
Don't have RabbitMQ handy? Spin one up with management UI in a single command: 
docker run -d --name rabbit -p 5672:5672 -p 15672:15672 rabbitmq:3-management
Then visit http://localhost:15672 (login: guest / guest).

Install

npm install runmq

Connect

Open a connection during application startup. RunMQ.start() resolves once the connection is fully ready.

import { RunMQ } from 'runmq';

const runMQ = await RunMQ.start({
  url: 'amqp://localhost:5672',
  reconnectDelay: 5000,        // ms between reconnect attempts (default 5000)
  maxReconnectAttempts: 5,     // give up after N attempts (default 5)
  management: {                // optional, but highly recommended
    url: 'http://localhost:15672',
    username: 'guest',
    password: 'guest',
  },
});
Why management matters. Providing it unlocks policy-based retry delays — meaning you can change attemptsDelay at runtime without re-declaring queues. Without it, RunMQ falls back to queue-based TTL (which works fine, just less flexible).

Set up your processors

A small but important detail: start your processors before you publish. Queues are created the first time a consumer subscribes — so a processor needs to be up for its queue to exist. 

// Processor 1: Email Service
await runMQ.process('user.created', {
  name: 'emailService',        // unique → isolated queue + DLQ
  consumersCount: 2,           // process up to 2 messages concurrently
  attempts: 3,                 // retry up to 3 times before DLQ
  attemptsDelay: 2000,         // wait 2s between retries
  usePoliciesForDelay: true,   // recommended (default: false)
}, async (message) => {
  console.log('EmailService received:', message.message);
  await sendEmail(message.message);
});

// Processor 2: SMS Service — same topic, separate queue
await runMQ.process('user.created', {
  name: 'smsService',
  consumersCount: 1,
  attempts: 5,
  attemptsDelay: 1000,
  usePoliciesForDelay: true,
}, async (message) => {
  console.log('SMSService received:', message.message);
  await sendSMS(message.message);
});

What's happening here:

  • name uniquely identifies the processor and gives it a dedicated queue + DLQ.
  • Pub/Sub is built in — both processors subscribe to user.created and each receive every message. One sends an email, the other sends an SMS, and they don't interfere.
  • Every processor gets its own retry policy, concurrency level, and delay configuration. Tune them per workload.
  • With management configured, you can change attemptsDelay later without re-declaring queues.

Publish a message

runMQ.publish('user.created', {
  userId: '123',
  email: 'user@example.com',
  name: 'John Doe',
});

One publish, every subscribed processor receives the message — independently and atomically.

What you've built

Publisher → Topic (user.created) ├→ Queue: emailService → DLQ: emailService_dlq └→ Queue: smsService → DLQ: smsService_dlq

Both processors are running. Each owns its queue and DLQ. If emailService throws, RunMQ retries it according to its own policy and (if exhausted) parks the message in emailService_dlq — without affecting smsService in any way.
