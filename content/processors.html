Core · Processors

Processors — the unit of work.

A processor is a named subscription to a topic. It owns a queue, a DLQ, a retry policy, and a concurrency level. Same topic, multiple processors? That's pub/sub, by construction.

Anatomy

await mq.process(topic, config, handler);
  • topic — the routing key. All processors with this topic receive every message.
  • config — sets the processor's name, retries, concurrency, optional schema.
  • handler — your async function. Throw to retry; resolve to ack.

Configuration

PropertyTypeDefaultWhat it does
namestring—Unique name. Each name gets a dedicated queue + DLQ.
consumersCountnumber1Concurrent in-flight messages (RabbitMQ prefetch).
consumersCountnumber1Concurrent consumers (independent AMQP channels). Each consumer keeps its own prefetch window, so total in-flight = consumersCount × prefetch.
prefetchnumber20Per-consumer prefetch count. Total unacked messages held by the processor is consumersCount × prefetch. New in 2.x.
attemptsnumber1Maximum delivery attempts before DLQ.
attemptsDelaynumber1000Milliseconds between retries.
usePoliciesForDelaybooleanfalseUse RabbitMQ policies for the retry delay TTL. Recommended.
messageSchemaMessageSchema—Optional schema for incoming-message validation.

The handler signature

type Handler<T> = (message: RunMQMessageContent<T>) => Promise<void>;

interface RunMQMessageContent<T> {
  message: T;                  // your payload
  meta: {
    id: string;                // unique message id
    publishedAt: number;       // unix millis
    correlationId: string;     // tracing id
  };
}

If the handler throws, RunMQ retries according to attempts / attemptsDelay. After the last attempt, the message goes to that processor's dedicated DLQ — other processors on the same topic are unaffected.

Worked example

interface UserCreated {
  userId: string;
  email: string;
  name: string;
}

await mq.process<UserCreated>('user.created', {
  name: 'emailService',
  consumersCount: 4,
  attempts: 3,
  attemptsDelay: 2_000,
  usePoliciesForDelay: true,
}, async ({ message, meta }) => {
  console.log('Processing message', meta.id, '(attempt logged by RunMQ)');
  await sendWelcomeEmail(message.email, message.name);
});

Concurrency vs scale

GoalHow
More throughput from one processIncrease consumersCount.
More throughput from a fleetRun multiple instances. They share the queue automatically.
Stricter orderingSet consumersCount: 1 and run a single instance.

Avoiding common mistakes

Start processors before publishing. Queues are created the first time a consumer subscribes. If you publish to a topic before any processor is up, you'll have nothing to receive that message — it will not be retroactively delivered.
Names must be unique. Two processors with the same name would collide on the same queue. RunMQ doesn't enforce this for you in the core package — pick stable, distinct names.

Pub/sub without fan-out logic

Two processors on the same topic? You've built pub/sub. Each processor owns its queue and gets every message — independently. No application-level fan-out, no orchestration. 

await mq.process('user.created', { name: 'emailService',  attempts: 3 }, sendEmail);
await mq.process('user.created', { name: 'analyticsSvc',  attempts: 5 }, track);

mq.publish('user.created', { /* payload */ });
// → both processors receive the message, atomically.
