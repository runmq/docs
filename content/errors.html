Errors & lifecycle.

RunMQ favors fail-fast. Misconfiguration throws at startup, not in production at the worst possible moment.

Startup errors

ScenarioBehavior
RabbitMQ unreachable at startupLogged via the configured logger, error is re-thrown so bootstrap fails fast.
publish() before connection is readyThrows Error('RunMQ is not connected').
Two processors with the same nameThrows at startup: Duplicate processor name: {name}.
@Processor class missing @ProcessMessageThrows at startup: No @ProcessMessage handler found in {ClassName}.
@Processor class with multiple @ProcessMessage methodsThrows at startup: Multiple @ProcessMessage handlers in {ClassName}.

Runtime errors

  • A handler that throws is treated as a transient failure and retried up to attempts times.
  • After the final attempt, the message is moved to that processor's DLQ. Other processors on the same topic are unaffected.
  • If a connection drops, in-flight unacked messages are re-delivered after reconnection (RabbitMQ semantics).

Custom logger interface

import { RunMQLogger } from 'runmq';

class CustomLogger implements RunMQLogger {
  log(message: string): void {
    // info-level
  }

  error(message: string, error?: any): void {
    // error-level
  }
}

Pass it to RunMQ.start(config, new CustomLogger()). In NestJS, use NestJS's logger — nestjs-runmq wires it automatically.

Graceful shutdown

async function shutdown() {
  server.close();          // stop accepting requests
  await mq.stop();         // drain handlers, close AMQP
  process.exit(0);
}

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
In NestJS, the module hooks into onModuleDestroy automatically. As long as app.enableShutdownHooks() is enabled, you don't need to write the wiring.
