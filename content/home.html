Why you need RunMQ

↻

Reliable retries

Failed messages are retried with configurable delays and limits, so transient errors don't take you down.

⌫

Dead Letter Queues

Once retries are exhausted, the message lands safely in a DLQ where you can inspect or replay it.

⇉

Pub/Sub, atomic

Publish a message once, every subscribed consumer gets it — no fan-out logic on your side.

▦

Isolated queues

Each processor owns its queue and its DLQ. One slow consumer can't drag the rest down.

✓

Schema validation

Optional JSON Schema validation (AJV) catches bad messages before they reach your business logic.

⇧⇩

Concurrent consumers

Scale horizontally (more instances) or vertically (more consumers per queue, via prefetch).

npm install runmq

A minute of code

Connect, declare a processor, publish a message. RunMQ takes care of the rest — retries, DLQ wiring, ack/nack semantics, reconnection. 

import { RunMQ } from 'runmq';

const runMQ = await RunMQ.start({
  url: 'amqp://localhost:5672',
  management: {
    url: 'http://localhost:15672',
    username: 'guest',
    password: 'guest',
  },
});

await runMQ.process('user.created', {
  name: 'emailService',
  consumersCount: 2,
  attempts: 3,
  attemptsDelay: 2000,
  usePoliciesForDelay: true,
}, async (message) => {
  await sendEmail(message.message);
});

runMQ.publish('user.created', {
  userId: '123',
  email: 'user@example.com',
  name: 'John Doe',
});
Using NestJS? Skip the wiring. nestjs-runmq gives you decorator-based processors, an injectable publisher, and full lifecycle integration — all in a few lines.
