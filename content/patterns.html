Patterns

Two patterns. One toolkit.

Almost every system that uses a queue ends up in one of two shapes — sometimes both. RunMQ models them with the same primitives, so you can start with one and grow into the other without a migration.

The two shapes

⚙

Background processing

Move slow or unreliable work off the request path. A worker pool drains a dedicated queue, with retries and a DLQ catching the ones that fail. Best when one logical worker owns one job type — emails, image transforms, exports, webhooks.

Read the guide →

⇄

Event bus (EDA)

Publish a fact once; many independent services react to it. Each consumer owns its queue, retries, and DLQ — full isolation. The backbone of decoupled microservices: services don't call each other, they react to events.

Read the guide →

How they relate

These aren't two different products. They're the same publish/process primitives arranged differently. Background processing is an event bus where exactly one consumer is subscribed to the topic. The moment a second service wants in, you add another processor — no refactor, no migration, no broker change.

Background processing: publish('email.send', …) ─► emailWorker (DLQ: emailWorker_dlq) Event bus: publish('user.created', …) ─► emailService (DLQ) ├─► analyticsService (DLQ) └─► notificationService(DLQ)

Choosing where to start

If you…Reach for
have one job type handled by one logical worker Background processing
have multiple services each reacting to the same event Event bus
aren't sure yet Start with background processing — promote later by adding processors
Both patterns share the same primitives. The only thing that changes between them is how many processors subscribe to the topic.

What they share

  • Per-processor isolation. Each processor owns its queue, retries, concurrency, and DLQ — failures stay contained.
  • Durable retries. Attempt counts live in the broker (x-death), so crashes never reset progress.
  • Replayable DLQs. Failed messages are inspectable and can be shovelled back to the original queue.
  • Schema validation at the edge. Reject bad payloads before they reach your handler.
  • Tunable delays. Per-processor TTLs you can change at runtime via RabbitMQ policies.
