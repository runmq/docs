Patterns

Patterns RunMQ fits naturally.

Two patterns cover the vast majority of use cases. They use the same primitives — RunMQ just lets you grow from one to the other without a migration.

1. Event-Driven Architecture (Event Bus)

Multiple services react to the same event independently. Each one owns its queue and its DLQ — full isolation, full autonomy.

Publisher → Topic (user.created) ├→ Queue: emailService → DLQ: emailService_dlq ├→ Queue: analyticsService → DLQ: analyticsService_dlq └→ Queue: notificationService → DLQ: notificationService_dlq

Why teams reach for this pattern:

  • One publish reaches every interested service — no fan-out logic in your app.
  • Each service tunes its own retries, concurrency, and delays.
  • Adding a new service is just subscribing to an existing topic — no upstream changes.
  • A failing consumer doesn't drag the others down; bad messages land in its DLQ.
  • Schema validation can stop invalid payloads before they reach your handlers.

Example

import { RunMQ } from 'runmq';

const mq = await RunMQ.start({ url: 'amqp://localhost:5672' });

// Email — short retries, low concurrency
await mq.process('user.created', {
  name: 'emailService',
  consumersCount: 2,
  attempts: 3,
  attemptsDelay: 2_000,
}, async (msg) => sendWelcomeEmail(msg.message));

// Analytics — many retries, fire-and-forget OK if it falls behind
await mq.process('user.created', {
  name: 'analyticsService',
  consumersCount: 4,
  attempts: 10,
  attemptsDelay: 5_000,
}, async (msg) => track('user_created', msg.message));

// Notifications — strict schema, every message is critical
await mq.process('user.created', {
  name: 'notificationService',
  consumersCount: 2,
  attempts: 5,
  attemptsDelay: 1_000,
  messageSchema: {
    type: 'ajv',
    failureStrategy: 'dlq',
    schema: { /* ... */ },
  },
}, async (msg) => pushNotification(msg.message));

mq.publish('user.created', { userId: 'u_123', email: 'a@b.com' });

2. Background Processing

A worker drains jobs from a dedicated queue, with retries and a DLQ for the ones that fail.

Publisher → Topic (email.send) → Queue: emailWorker → DLQ: emailWorker_dlq

Why this works well:

  • Run multiple workers in parallel for high throughput.
  • Failures are captured in the DLQ where you can inspect or replay them.
  • Schema validation keeps malformed jobs from breaking your worker.
  • If your needs grow, this pattern transforms into an Event Bus by simply adding more processors to the same topic — no migration required.
await mq.process('email.send', {
  name: 'emailWorker',
  consumersCount: 8,           // scale vertically inside one process
  attempts: 5,
  attemptsDelay: 30_000,
  usePoliciesForDelay: true,   // tune later without re-declaring queues
}, async (job) => {
  await transport.send(job.message.to, job.message.subject, job.message.body);
});

// Anywhere in your app:
mq.publish('email.send', {
  to: 'customer@example.com',
  subject: 'Welcome',
  body: '...',
});

Choosing between them

If you…Reach for
have one job type handled by one logical worker Background processing
have multiple services each reacting to the same event Event Bus
aren't sure yet Start with background processing — promote later by adding processors
Both patterns share the same primitives. The only thing that changes between them is how many processors subscribe to the topic.

Mixing strategies per processor

Because each processor has its own queue, retry policy, and concurrency, you can mix strategies on a single topic. For example, the email consumer might retry quickly for 5 minutes, while the archive consumer retries patiently over an hour. Both subscribe to the same event; neither blocks the other.
