Patterns

Background processing done right.

Move slow, expensive, or unreliable work off the request path. A worker pool drains a dedicated queue, retries on failure, and parks unrecoverable jobs in a DLQ. The simplest pattern to start with — and the one most applications need first.

Quick start

import { RunMQ } from 'runmq';

const mq = await RunMQ.start({ url: 'amqp://localhost:5672' });

await mq.process('email.send', {
  name: 'emailWorker',
  consumersCount: 8,           // 8 messages in flight per process
  attempts: 5,
  attemptsDelay: 30_000,       // 30s between retries
  usePoliciesForDelay: true,   // tune the delay later without redeploying
}, async (job) => {
  await transport.send(job.message.to, job.message.subject, job.message.body);
});

// Anywhere in your app — an HTTP handler, a cron job, another service:
await mq.publish('email.send', {
  to: 'customer@example.com',
  subject: 'Welcome',
  body: '...',
});

That's the whole thing. The queue, retries, DLQ, and re-injection topology are created automatically.

Tuning

Concurrency: consumersCount

The number of messages a single worker process handles in parallel. Pick it based on the bottleneck:

BottleneckSuggested consumersCount
I/O-bound (HTTP, DB, disk)16–64. Async I/O barely uses CPU.
CPU-bound (image, crypto)1–2 per CPU core. More just causes contention.
Rate-limited external APIMatch the rate limit. Use 1 if the API allows 1 RPS.
Memory-heavy (large blobs)Low. Each in-flight message holds memory.

Retries: attempts and attemptsDelay

  • attempts — max times RunMQ hands a message to your handler before sending it to the DLQ.
  • attemptsDelay — milliseconds the message waits between failure and the next attempt.

Retries are for transient failures. They don't fix bugs.

Failure modeSuggested settingsReasoning
Transient network failures attempts: 5, attemptsDelay: 5_000–30_000 Most blips clear in under a minute.
Rate-limited APIs (429) attempts: 3–5, attemptsDelay: 60_000+ Retrying faster than the API allows just keeps you rate-limited. To honor exact Retry-After headers, do it inside the handler.
Outbound webhooks attempts: 10+, minutes-to-hours delay Customer endpoints may be down for maintenance.
Validation / business-rule failures attempts: 1 A malformed payload won't become valid on retry.
Use usePoliciesForDelay: true in production. By default attemptsDelay is baked into the retry queue at declare time, so changing it later means deleting and re-declaring the queue. With policies on, the delay lives in a RabbitMQ policy you can edit at runtime — handy when a downstream is degraded and you want to dial the delay up without touching queues.

Schema validation

Stop bad jobs from ever reaching your handler. RunMQ supports an Ajv-based schema with two failure strategies:

await mq.process('image.resize', {
  name: 'imageWorker',
  consumersCount: 4,
  attempts: 3,
  attemptsDelay: 10_000,
  messageSchema: {
    type: 'ajv',
    failureStrategy: 'dlq',          // or 'retry' for fixable issues
    schema: {
      type: 'object',
      required: ['url', 'width'],
      properties: {
        url: { type: 'string', format: 'uri' },
        width: { type: 'integer', minimum: 16, maximum: 4096 },
      },
    },
  },
}, async (job) => { /* job.message is validated */ });

'dlq': malformed messages skip retries entirely — validation failures don't fix themselves over time. 'retry': appropriate when schemas can drift (e.g., briefly during a deploy that introduces a new validator).

Idempotency

Retries mean any handler may run more than once (e.g., crash after the work, before ack). Make handlers idempotent: 

await mq.process('payment.charge', { name: 'payments', attempts: 5, attemptsDelay: 10_000 }, async (job) => {
  const { idempotencyKey, amount, customerId } = job.message;
  if (await db.payments.findByKey(idempotencyKey)) return;       // already done
  await stripe.charges.create({ amount, customer: customerId, idempotency_key: idempotencyKey });
  await db.payments.recordKey(idempotencyKey);
});

The producer generates a stable key per logical operation; the handler checks "did I already do this?" before doing it. The cheapest insurance against double-processing.

Operating the DLQ

When a job exhausts its retries it lands in _runmq_dlq_{name}. RunMQ's DLQ is bound back to the router exchange, so replay is just moving messages from the DLQ back to the processor queue — and RunMQ Pulse is the dashboard built for it. Pulse browses DLQ contents, surfaces the failure reason from x-death, and replays messages back to the original queue with one click.

  • Inspect first. A flood of DLQ'd messages usually points to a bug or a downstream incident — replay won't help until you fix the root cause.
  • Triage. Sort by error class. The fix for "auth token expired" is different from "payload too large".
  • Replay deliberately. Once the underlying issue is resolved, replay. Retry counts reset.

Pulse is optional — anything Pulse does, you can do with the management UI or the shovel plugin. It just removes the friction so DLQs stay something you actually look at.

Anti-patterns

Don'tWhy
Use background jobs for work the user is waiting on If the user is staring at a spinner, you need a faster path — not a queue.
Set attempts to 1000 Retries hide bugs. If a job is failing 50 times, the 51st won't fix it.
Make handlers non-idempotent Charge a card twice and your customer will notice.
Stuff giant payloads in the message RabbitMQ isn't a blob store. Pass a reference ({ s3Key: '…' }) instead.
Mix unrelated jobs in one queue One processor = one job type. Mixing means a slow type blocks the rest, and tuning is a compromise.

Deep dive: how the pattern works

What background processing is

A queue used to defer work that doesn't need to happen synchronously. The handler that triggered the work enqueues a job and returns immediately; a separate worker process picks the job up and runs it on its own time.

HTTP request ─► handler ─► publish('email.send', …) ─► 200 OK │ ▼ queue: emailWorker (N consumers) │ ┌───────────┴────────────┐ ▼ ▼ retries (TTL) DLQ on exhaustion

When to reach for it

PropertyWhy a queue helpsExamples
Slow Free the request thread; latency-sensitive APIs stay fast. Reports, exports, transcoding
Expensive Cap concurrency at the consumer; backpressure via queue depth. Image processing, ML inference
Unreliable Automatic retries with TTL-based delays. Third-party APIs, outbound webhooks
Bursty The queue smooths spikes; workers drain at their own pace. Signup floods, batch imports
Optional Decouple user response from side-effects. Emails, notifications, analytics
Rule of thumb: if the user wouldn't notice (or wouldn't mind) the work happening 200ms — or 30s — later, it belongs in a background job.

Why it matters in microservices

Every synchronous call between services is a coupling. If A calls B which calls C, A's response time is the sum of all three, and any one being down breaks the whole chain. Background processing breaks the chain in three ways:

  • Latency isolation. The user-facing service stays fast; the hand-off is a sub-millisecond enqueue.
  • Failure isolation. If the worker is down, jobs accumulate in the queue. The producer never sees errors.
  • Scaling independence. The web tier scales by RPS; the worker tier scales by queue depth — different metrics, different machines.
