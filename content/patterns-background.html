Patterns

Background processing done right.

Move slow, expensive, or unreliable work off the request path. A worker pool drains a dedicated queue, retries on failure, and parks unrecoverable jobs in a DLQ for inspection or replay. It's the simplest pattern to start with — and the one most applications need first.

What it is

Background processing is the use of a queue to defer work that doesn't need to happen synchronously. Instead of doing the work inside the HTTP handler that triggered it, the handler enqueues a job and returns immediately. A separate worker process — or a pool of them — picks the job up and runs it on its own time.

HTTP request ─► handler ─► publish('email.send', …) ─► 200 OK │ ▼ queue: emailWorker (N consumers) │ ┌───────────┴────────────┐ ▼ ▼ retries (TTL) DLQ on exhaustion

RunMQ's primitives map onto this pattern directly: a single processor binds to a topic, declares a queue, and runs your handler with retries and a DLQ wired in. No infrastructure to write — just a function.

When to use it

Reach for background processing whenever a piece of work has any of these properties:

PropertyWhy a queue helps
Slow — takes longer than a request should Free the request thread. Latency-sensitive APIs stay fast under load.
Expensive — CPU, memory, or external rate-limited APIs Cap concurrency at the consumer. Apply backpressure naturally via queue depth.
Unreliable — third-party APIs, flaky networks Automatic retries with exponential-style delays via TTL policies.
Bursty — peaks far above sustained load The queue smooths spikes. Workers drain at their own pace.
Optional — best-effort work the user shouldn't wait on Decouple the user's response from the side-effect.

Concrete examples

  • Transactional email — welcome emails, password resets, receipts.
  • Notifications — push, SMS, in-app, with rate limits and retry semantics.
  • Media processing — image resizing, video transcoding, thumbnail generation.
  • Webhook delivery — outbound webhooks to customers, with retries on 5xx.
  • Reports & exports — long CSV/PDF generation, then notify the user when ready.
  • Index updates — re-indexing search documents after a write.
  • Periodic sync — pulling third-party data on a schedule.
  • Cleanup jobs — deleting expired sessions, archiving old rows.

Why it matters in microservices

In a microservice architecture, every synchronous call between services is a coupling — and a liability. If service A calls service B which calls service C, A's response time is the sum of all three, and any one of them being slow or down breaks the whole chain.

Background processing breaks that chain in three important ways:

  • Latency isolation. A user-facing service stays fast even when a downstream worker is slow. The hand-off is a sub-millisecond enqueue — not an unbounded synchronous call.
  • Failure isolation. If the worker is down, jobs accumulate in the queue. The producer doesn't see errors; the queue absorbs them. When the worker recovers, it drains the backlog. No retries-in-the-request, no cascading failures.
  • Scaling independence. The web tier scales by RPS; the worker tier scales by queue depth. They use different metrics, different machines, and often different deploy cadences. You can scale the part of your system that's actually bottlenecked without over-provisioning the rest.
Rule of thumb: if the user wouldn't notice (or wouldn't mind) the work happening 200ms — or 30s — later, it belongs in a background job. Anything you can move off the request path makes the request path more reliable.

The minimum viable setup

import { RunMQ } from 'runmq';

const mq = await RunMQ.start({ url: 'amqp://localhost:5672' });

await mq.process('email.send', {
  name: 'emailWorker',
  consumersCount: 8,           // 8 messages in flight per process
  attempts: 5,
  attemptsDelay: 30_000,       // 30s between retries
  usePoliciesForDelay: true,   // tune the delay later without redeploying
}, async (job) => {
  await transport.send(job.message.to, job.message.subject, job.message.body);
});

// Anywhere in your app — an HTTP handler, a cron job, another service:
mq.publish('email.send', {
  to: 'customer@example.com',
  subject: 'Welcome',
  body: '...',
});

That's the whole thing. The queue, retries, DLQ, and re-injection topology are created automatically.

Tuning for your workload

Concurrency: consumersCount

This is the number of messages a single worker process will handle in parallel. Pick it based on what bottlenecks your handler:

BottleneckSuggested consumersCount
I/O-bound (HTTP, DB, disk)16–64. Async I/O barely uses CPU.
CPU-bound (image, crypto)1–2 per CPU core. More just causes contention.
Rate-limited external APIMatch the rate limit. Use 1 if the API allows 1 RPS.
Memory-heavy (large blobs)Low. Each in-flight message holds memory.

Retries: attempts and attemptsDelay

Retries paper over transient failures — network blips, brief downtimes, rate-limit responses. They don't fix bugs, and they don't help with permanent errors. Pick numbers that reflect the real failure mode:

  • Transient network failures: attempts: 5, attemptsDelay: 5_000–30_000.
  • Rate-limited APIs: fewer attempts, longer delays. Better: respect Retry-After headers.
  • Webhook delivery: many attempts (10+) over hours; customers expect this.
  • Validation-driven failures: attempts: 1. Retrying invalid input never helps.
Use usePoliciesForDelay: true in production. It lets you change retry delays at runtime without re-declaring queues. During an incident — say, a downstream is degraded — you can dial the delay up to 5 minutes from a dashboard, then back down when it recovers.

Schema validation

Stop bad jobs from ever reaching your handler. RunMQ supports an Ajv-based schema with two failure strategies: 

await mq.process('image.resize', {
  name: 'imageWorker',
  consumersCount: 4,
  attempts: 3,
  attemptsDelay: 10_000,
  messageSchema: {
    type: 'ajv',
    failureStrategy: 'dlq',          // or 'retry' for fixable issues
    schema: {
      type: 'object',
      required: ['url', 'width'],
      properties: {
        url: { type: 'string', format: 'uri' },
        width: { type: 'integer', minimum: 16, maximum: 4096 },
      },
    },
  },
}, async (job) => { /* job.message is validated */ });

With 'dlq', malformed messages skip retries entirely and land in the DLQ — because validation failures don't fix themselves over time. 'retry' is appropriate when schemas can drift (e.g., you just deployed a new validator and there's a brief overlap with old producers).

Operating the DLQ

When a job exhausts its retries, it lands in _runmq_dlq_{name} for inspection. RunMQ's DLQ is bound back to the router exchange, so replay is just moving messages from the DLQ back to the processor queue — no special tooling needed beyond RabbitMQ's built-in shovel or the management UI.

  • Inspect first. A flood of DLQ'd messages usually points to a bug or a downstream incident — replay won't help until you fix the root cause.
  • Triage. Sort DLQ entries by error class. The fix for "auth token expired" is different from "payload too large".
  • Replay deliberately. Once the underlying issue is resolved, shovel back. The handler sees a fresh delivery; retry counts reset.

Anti-patterns

Don'tWhy
Use background jobs for work the user is waiting on If the user is staring at a spinner, you need a faster path — not a queue. Use a queue for fire-and-forget work, or pair it with a "ready" notification (email, websocket).
Set attempts to 1000 Retries hide bugs. If a job is failing 50 times, the 51st attempt isn't going to fix it.
Make handlers non-idempotent Retries mean any successful handler may run more than once (e.g., crash after success, before ack). Charge a card twice and your customer will notice.
Stuff giant payloads in the message RabbitMQ isn't a blob store. Put a reference ({ s3Key: '…' }) in the message and load the blob in the handler.
Mix unrelated jobs in one queue One processor = one job type. Mixing means a slow job type blocks the rest, and tuning is a compromise.

Idempotency in three lines

await mq.process('payment.charge', { name: 'payments', attempts: 5, attemptsDelay: 10_000 }, async (job) => {
  const { idempotencyKey, amount, customerId } = job.message;
  if (await db.payments.findByKey(idempotencyKey)) return;       // already done
  await stripe.charges.create({ amount, customer: customerId, idempotency_key: idempotencyKey });
  await db.payments.recordKey(idempotencyKey);
});

The producer generates a stable key per logical operation. The handler checks "did I already do this?" before doing it. This is the single most important habit for queue-based systems — and the cheapest insurance against double-processing.

Growing into an event bus

When a second team wants to do something with email.send events — say, count them for analytics — you don't need to migrate. You add a second processor on the same topic with its own queue. The two consumers run independently; failures in analytics don't affect email delivery.
