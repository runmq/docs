Connection & lifecycle.

RunMQ wraps a single AMQP connection and exposes it as a small, well-behaved client. You start it once, share it across the app, and let it manage reconnects, heartbeats, and graceful shutdown.

Starting RunMQ

The RunMQ.start() async factory opens the connection and resolves only when the broker handshake is complete. After it resolves, you can declare processors and publish.

import { RunMQ } from 'runmq';

const mq = await RunMQ.start({
  url: 'amqp://guest:guest@localhost:5672',
  reconnectDelay: 5_000,
  maxReconnectAttempts: 5,
  management: {
    url: 'http://localhost:15672',
    username: 'guest',
    password: 'guest',
  },
});

Reconnect behavior

  • reconnectDelay — wait between reconnect attempts.
  • maxReconnectAttempts — caps how many times RunMQ will retry before giving up.
  • If the connection drops mid-flight, in-flight handlers complete (or fail and retry) according to RabbitMQ's ack semantics. RunMQ does not duplicate work it has already acked.

Health checks

Use isActive() for liveness. It returns true when the underlying AMQP channel is open.

app.get('/healthz', (req, res) => {
  res.status(mq.isActive() ? 200 : 503).json({
    rabbitmq: mq.isActive() ? 'up' : 'down',
  });
});

Graceful shutdown

On SIGTERM, stop publishing first, then close the client. Consumers drain in-flight messages cleanly.

async function shutdown() {
  // 1. stop accepting new work
  server.close();

  // 2. let in-flight messages finish, then close AMQP
  await mq.stop();
  process.exit(0);
}

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
Don't share connections across processes. Each Node.js process should call RunMQ.start() once on its own. RunMQ is single-instance per process — fork it for multi-core scaling.
