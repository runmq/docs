Behind the scenes

How RunMQ works internally.

RunMQ is a thin, opinionated layer over RabbitMQ. The whole library boils down to a small number of exchanges, per-processor queues, and a clever use of dead-letter exchanges to do retries with delay. This page walks the topology end-to-end, with the actual broker objects RunMQ creates.

The big picture

A single router exchange is the entry point for everything you publish. RunMQ creates one queue per processor (with its own DLQ) and a per-processor retry-delay queue that uses RabbitMQ's TTL + dead-letter mechanism to "park" failed messages until the delay expires, then re-injects them into the original queue.

The objects RunMQ creates

For every processor named name, RunMQ asserts the following on the broker:

ObjectNameConfigured with
Topic exchange_runmq_routerShared. The single entry point for all publishes.
Topic exchange_runmq_dead_letter_routerShared. Used to route rejected messages by processor name.
Processor queue{name}Durable. DLX = _runmq_dead_letter_router, DLRK = {name}.
Retry-delay queue_runmq_retry_delay_{name}Durable. DLX = _runmq_router. TTL = attemptsDelay (queue arg or policy).
Dead-letter queue_runmq_dlq_{name}Durable. DLX = _runmq_router, DLRK = {name} — so messages can be replayed.
Metadata_runmq_metadata_{name}Stored via the management API for dashboards (max retries, timestamps).

The bindings

Bindings determine how a published message reaches a queue. RunMQ wires four of them per processor:

_runmq_router ── topic={topic} ────────────► queue: {name} _runmq_router ── topic={name} ─────────────► queue: {name} (retry re-injection) _runmq_dead_letter_router ── topic={name} ─────────────► queue: _runmq_retry_delay_{name} _runmq_dead_letter_router ── topic=_runmq_dlq_{name} ──► queue: _runmq_dlq_{name}

The two bindings on the processor queue are deliberate:

  • By topic — handles the publish path: publish('user.created', …) reaches every processor subscribed to user.created.
  • By processor name — handles the retry path: when a delayed message expires its TTL, it's dead-lettered back to _runmq_router with the routing key set to the processor name. Only that one processor's queue picks it up — siblings don't see retries that aren't theirs.

Lifecycle of a message

1. Publish

Your publish(topic, payload) call writes a JSON-encoded envelope to the AMQP buffer for the _runmq_router exchange with routing key = topic. Nagle's algorithm is disabled on the socket so messages flush immediately.

2. Routing & fan-out

The router exchange copies the message to every bound processor queue. Two processors on user.created means two independent copies — that's pub/sub, atomic at the broker level. No application-side fan-out, no risk of "delivered to A but not B."

3. Consume

Each processor's consumer pulls the message and runs it through a small chain of decorators (more on that below). If your handler resolves, the message is acked and removed.

4. Failure → delay

If your handler throws, the message is rejected with requeue=false. RabbitMQ's dead-letter machinery kicks in:

  • Processor queue's DLX is _runmq_dead_letter_router, DLRK = processor name.
  • That exchange has a binding to _runmq_retry_delay_{name}, so the message lands there.
  • The retry queue has a TTL — either as a queue argument (messageTtl) or via a RabbitMQ policy. The message sits idle until the TTL expires.

5. Re-injection

When TTL expires, RabbitMQ dead-letters the message again. The retry queue's DLX is _runmq_router — and because messages keep their original routing key when dead-lettered (which RunMQ relies on), the message is re-routed back to the same processor queue. The handler sees it again.

6. Final failure → DLQ

RabbitMQ writes an x-death header that records every reject + timestamp. RunMQ counts how many times this message has been rejected for the processor's queue. Once the count reaches attempts, the message is not sent through the delay queue again — it's published directly to _runmq_dlq_{name} via the dead-letter exchange and acked. There it waits for inspection or replay.

The DLQ can replay. The DLQ's own DLX points back to the router with routing key = processor name. If you want to replay a DLQ'd message, you can move it back from the DLQ and it will land in the original processor queue exactly as a fresh delivery would.

Counting attempts: the x-death header

RabbitMQ stamps each dead-lettered message with an x-death header — an array of records describing each reject event. RunMQ reads it to decide whether to retry or route to the DLQ: 

private getRejectionCount(message: RabbitMQMessage): number {
  const xDeath = message.headers?.["x-death"];
  if (!Array.isArray(xDeath)) return 1;
  const deathRecord = xDeath.filter(e => e && e.reason == 'rejected')[0];
  return deathRecord ? deathRecord.count + 1 : 1;
}

This is what powers the "retry up to N times" semantics — RunMQ doesn't track attempts itself, the broker does. That makes retries durable across restarts: kill the consumer mid-flight, restart it, and the retry counter is still right.

Two flavors of retry delay

ModeHow TTL is setTrade-off
usePoliciesForDelay: false (default) Queue argument (messageTtl) baked into the queue at declare time. Simple, no management API needed. Changing the delay later requires deleting and re-declaring the queue.
usePoliciesForDelay: true (recommended) RabbitMQ policy applied via the management HTTP API. Delay can be changed at runtime without queue surgery. Requires the management config block.

Policies are why RunMQ Pulse can dial retry delays live during an incident. The dashboard updates the policy; the broker reflows the TTL; in-flight retries pick up the new delay.

The consumer pipeline

Each delivery goes through a short chain of decorators before it ever reaches your handler. The chain is built per-message in RunMQConsumerCreator.runProcessor: 

new RunMQExceptionLoggerProcessor(
  new RunMQSucceededMessageAcknowledgerProcessor(
    new RunMQFailedMessageRejecterProcessor(
      new RunMQRetriesCheckerProcessor(
        new RunMQFailureLoggerProcessor(
          new RunMQBaseProcessor<T>(yourHandler, config, deserializer)
        ),
        config, DLQPublisher, logger
      )
    )
  ),
  logger
).consume(rabbitmqMessage);

From outside in, each decorator owns a single concern:

LayerResponsibility
ExceptionLoggerCatches anything that escapes the chain and logs it.
SucceededMessageAcknowledgerAcks the message after the inner chain returns success.
FailedMessageRejecterNacks (with requeue=false) on failure → triggers the DLX path.
RetriesCheckerReads x-death, decides retry vs. DLQ, publishes to DLQ when exhausted.
FailureLoggerLogs the handler exception with context.
BaseProcessorDeserializes the payload and calls your handler.

This is a small price to pay (≈6 object allocations per message) for code where each concern is independently testable and swappable. It's also why retries, acks, and DLQ routing never accidentally interleave — they live in separate, ordered steps.

Naming conventions

Everything RunMQ creates is prefixed with _runmq_, so it's easy to find — and easy to leave alone — in the RabbitMQ management UI:

_runmq_dlq_{name}Per-processor DLQ.

What this design buys you

  • Atomic pub/sub with isolation. One publish, N independent consumer queues — failures stay contained.
  • Durable retries. Attempt counts live in the broker (x-death), not in your process. Crashes don't reset retries.
  • Tunable delays without redeploys. Policies let you change retry behavior at runtime.
  • Replay-friendly DLQs. The DLQ is itself dead-letter-bound back to the router, so replay is just "shovel back."
  • Discoverable topology. Predictable naming + metadata stored in the broker means dashboards (like Pulse) can describe your system without seeing your code.
RunMQ doesn't reinvent broker primitives — it composes them. Every "feature" you see is the right combination of an exchange, a binding, a TTL, and a header.
